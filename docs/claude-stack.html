How one domain runs - a feature or a bug fix
the plan comes from one of two
solution-designera feature → tasks + contracts
issue-diagnosera bug → a decomposed fix plan ⇄ evidence-gatherer ×N
→
implementers · in parallel · one task each
implementer · task A implementer · task B implementer · task C
→
verifier once all parts land: vs plan + code quality
← ·········  gaps return to the implementers as a punch-list, and re-verify, until the verifier signs off. The main session runs the fan-out - the vertical's seats never dispatch agents; the two diagnosers are the one exception, each dispatching a read-only evidence-gatherer.
01

The team

Role down the side, domain across the top. The integration, diagnose and analysis rows span every stack; the design, build and verify rows get one seat per domain. The architecture capture is a skill now (/project-architecture-analyzer), run in the main session - there is no dispatched architect seat.
