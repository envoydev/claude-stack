# Domain Trio Protocol - One Stack's Design, Build, Verify Vertical

The execution protocol for the single-stack modes (`domain_trio`, `fanout_domain_trio`). By the time this file is read, the scoping verdict has already named the stack - a user hint ('frontend only', 'just the API') pins it the same way. The seats are resolved from the installed roster by the naming pattern `<stack>-solution-designer` / `<stack>-implementer` / `<stack>-verifier` (the generated capabilities rule lists what this project actually has). Everything technology-specific - conventions, patterns, what a good design or a passing verify looks like - lives in the seats and the skills they load, never here.

DELEGATED vs INLINE is picked once, before DESIGN (the shared policy in `execution-modes.md`): DELEGATED dispatches every seat; INLINE does the same three steps in-session - design, build the tasks in the designer's order, verify against the plan.

## Steps

1. **DESIGN** - dispatch the stack's solution designer with the clarified requirement (the clarify gate already ran in the skill body - never design against a guessed one). For a diagnosed bug the plan comes from the diagnosis instead (`issue-investigation.md`). Either way the output shape is the same: the architecture or the proven root cause, the test strategy, and a decomposition of independent tasks with contracts (each task's boundary and what it hands off or depends on). Gate the returned plan before anything builds - invoke `project-verify-plan` (the Skill tool, never its passes replayed from memory) and run its four passes in-session, recording the four per-pass verdicts (`risk / scope / edges / soundness`) in the ledger; a failed pass is a scoped re-brief to the designer - then STOP: present the gated plan and end the turn, and build only on the user's explicit word (the skill body's plan review stop). A blanket dispatch opt-in in the ask is NOT that word, and neither is a run-the-whole-flow or completion-token instruction - only an explicit review waiver ('run without plan review') skips the stop, recorded as `plan_review: waived` in the ledger. Never fan out against an unaudited, unreviewed, or unapproved plan.
2. **BUILD (fan-out)** - dispatch one implementer per independent task, all in parallel, from the main session. Fan out every task the designer marked independent, not a token two - the runtime bounds real concurrency; your job is to expose all of it. Hand each implementer exactly one task plus its contract; it never touches anything outside its boundary, which is what keeps the parallel runs collision-free. Dispatch each with the `implementer_model` the designer stamped on the task card, passed as the per-dispatch model override; the seat's frontmatter pin is the fallback when a card carries none (full rule in `model-routing.md`).
3. **VERIFY (fan-in)** - once every task lands, dispatch the stack's verifier over the assembled whole, never over a single task. On a punch-list, route each item back to the implementer that owns it, **resumed with a scoped brief** - the failing assertion or check plus the file and symbol to touch, not a fresh full-context re-brief (a full re-dispatch roughly doubles the cell's cost for a one-line fix) - then re-dispatch the verifier. The verifier's independent checks (build, the test suite, the contract-seam probe) are parallelizable - run them concurrently where the seat supports it. Loop until sign-off, but bound the loop: **two implementer fix rounds maximum**; still failing after that, stop and escalate to the user rather than looping to sign-off forever.

## Bookkeeping

- **Progress ledger.** The run's ledger is the same durable file `agent-output-protocol.md` formats - every task and its status, each verify pass, and the contract_version when the run sits under a frozen cross-domain contract - so a mid-run compaction resumes without re-deriving what already landed.
- **File hand-off, per-task review.** Each implementer gets its task + contract as a written brief and returns a written report; glance at each task's diff the moment it returns - not after the whole fan-out, and not by serializing the implementers. The verifier still runs once over the assembled whole; the early glance only catches a contract breach before it compounds.
- **Status routing.** Route on each implementer's closing status: DONE -> hand to the verifier; DONE_WITH_CONCERNS -> carry the flag to the verifier; NEEDS_CONTEXT -> close the gap via the designer, then re-dispatch; BLOCKED -> sequence after the blocking task; BLOCKED_CONTRACT_CHANGE -> pause the affected work and run the contract-change protocol (`contract-protocol.md`). The routing lives here; the status definitions and structured shapes are `agent-output-protocol.md`.
- **Memory hygiene.** The serena handoff notes (`<feature>__<contract_version>__<seat>`) live in the gitignored `.serena/memories/` and outlive a branch switch - a seat reads only the note matching its `<feature>__<contract_version>`, never a stale one. After the verifier signs off, purge the run's notes (serena `delete_memory` on each `<feature>__*`) so they never bleed into a later run that reuses the workspace.

## Rules

- The main session is the only orchestrator for the whole vertical. The domain seats carry no Agent tool - the fan-out stays flat, and no seat is ever told to dispatch another.
- Fan out only the tasks the designer marked independent, and respect the contracts it drew between them - two implementers touching the same boundary is the collision the contracts exist to prevent.
- The architecture docs are not refreshed by this flow. The fit verdict (extend / refactor first / isolate) is the designer's call from the committed map; reconciling the docs after a structural change is a deliberate capture run (`project-architecture-analyzer` / `project-architecture-quality-loop`), never a step here.
