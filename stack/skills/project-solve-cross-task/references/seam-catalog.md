# Seam Catalog - what turns a 'local' task cross-domain

The scoping pass walks these stack-keyed seams before calling a task local. The scoping miss - a task declared local that is actually coupled through a hidden seam - is what detonates mid-build, so a discovered shared-interface edit is itself the escalation signal, not just an obviously multi-stack ask. Walk the section for the stack the touched module lives in.

- **.NET / ASP.NET / EF Core:** a request/response DTO or public-contract edit ripples to every serializer, API client, and versioned consumer - flag it a seam change, not a local edit. Any entity/`DbContext` change drags a migration: is it destructive (column drop/rename), does it need a data backfill, is `Up()` idempotent and guarded, will it lock a large table on deploy. A method going async is async-all-the-way - name the caller chain that must change and thread `CancellationToken`. Touching a DI-registered service crosses its lifetime (a singleton capturing a scoped dep is a captive dependency); a shared `IValidator`/FluentValidation rule or an `[Authorize]` policy edit hits every endpoint using it. A write-path change invalidates whatever cache or read-model mirrors it, and an `AsNoTracking` read path will not observe a change-tracked write.
- **WPF (the other C# stack):** UI touched off the dispatcher thread needs `Dispatcher.Invoke`/`BeginInvoke`; a `PropertyChanged`/`CollectionChanged` handler or subscription never detached leaks the view; a `DependencyProperty` or binding-path change breaks bindings silently at runtime, not compile time.
- **Angular:** a `providedIn: 'root'` service edit reaches every injector; an `@Input()`/`@Output()` shape change ripples to every parent template binding it; a subscription without teardown (`takeUntilDestroyed` or the async pipe) leaks, and an `OnPush` component will not re-render when an input object is mutated in place rather than replaced. A shared model interface or reactive-form control shape consumed across feature modules is the cross-module escalation signal, as is an NgRx/signal-store slice or a route guard/resolver read by call sites the task never names.
- **Mobile (Ionic/Capacitor):** a plugin or permission change needs native config (iOS `Info.plist`, Android manifest) and can diverge iOS vs Android - platform parity is inside the scope; Ionic page-lifecycle hooks (`ionViewWillEnter`) fire differently from `ngOnInit` under cached page reuse.
- **Data / SQL:** a migration that locks or rewrites a large table, a non-idempotent `Up()` lacking an existence guard, an index change that shifts query plans, or a view/stored-proc consumed by multiple call sites - each turns a 'small' schema task into a coordinated one.
- **Tests are blast radius, not an afterthought:** the tests that pin the current behavior must change when behavior changes, and a test that never exercises the changed path stays falsely green - name both the tests that will break and the coverage gap that will hide the regression.

Never claim impact not located - every 'X breaks' names the symbol it came from, and unknown is reported as unknown, not guessed into certainty. Do not shrink the task to what is easy to analyze: if it implies a migration, a breaking change, or a slow investigation, say so even when it is unwelcome.
